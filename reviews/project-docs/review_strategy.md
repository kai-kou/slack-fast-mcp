# レビュー結果: ① 戦略（Why）

**レビュアー**: 戦略レビュアー
**対象ドキュメント**: requirements.md / architecture.md / slack-app-setup.md
**レビュー日**: 2026-02-10

---

## 総合評価: B+（良好、改善余地あり）

3つのドキュメント群は、プロジェクトの「なぜ」と「何を」を概ね適切に記述しているが、**中長期の戦略的視点**と**組織全体への波及効果**に関していくつかの欠落がある。

---

## 1. 目的整合性の検証

### 1.1 ユーザー要件と技術選定の整合性: ◎ 優秀

| ユーザー要件 | 技術的解決策 | 整合性 |
|---|---|---|
| 実行速度最優先 | Go言語・シングルバイナリ | ◎ 的確な選定 |
| Cursorが扱いやすい | MCP Server実装 | ◎ ネイティブサポート活用 |
| インストールコスト低 | シングルバイナリ配布 | ◎ ランタイム不要 |
| プロジェクト別設定 | ローカル設定ファイル | ◎ 標準的なアプローチ |
| ユーザーも利用 | CLIモード併設 | ○ 合理的だが優先度の議論が必要 |

**評価**: ユーザー要件から技術選定までのトレーサビリティが明確で、要件分析セクション（requirements.md §2）の構造が秀逸。各要件の「→ 技術的解決策」の流れが読み手に伝わりやすい。

### 1.2 懸念事項

- **CLIモードの戦略的位置づけが曖昧**: MCP Serverが主目的なのか、CLIも同等に重要なのかが不明確。「ユーザー(人)も利用することを想定したい」は、MVP段階で本当に必要か？CLIモードの開発工数がMCP Serverの品質向上を圧迫するリスクがある。
- **「いい感じのツールになったら公開」の曖昧さ**: 公開判断の基準が不在。何をもって「いい感じ」とするのか、定量的な基準（例: テストカバレッジ80%以上、ドキュメント完備等）が必要。

---

## 2. 組織戦略との整合性

### 2.1 指摘事項

| # | 指摘 | 重要度 | 説明 |
|---|---|---|---|
| S-1 | **OSS公開の戦略的意図が不明** | 高 | 「広く利用してほしい」とあるが、なぜOSS公開するのか（ブランディング？採用？コミュニティ貢献？）が記述されていない。目的が曖昧だと、公開後のメンテナンスコスト判断ができない |
| S-2 | **競合・既存ツールとの差別化が未記述** | 高 | Slack MCP Serverは既に複数存在する（公式のSlack MCP等）。なぜ自作するのか、既存ツールとの差別化ポイント（速度？軽量さ？）を明記すべき |
| S-3 | **チーム展開戦略が未定義** | 中 | 「チームメンバーに展開」とあるが、展開計画（段階的ロールアウト？全員一斉？）、サポート体制、フィードバック収集方法が未定義 |
| S-4 | **成功指標（KPI）が不在** | 中 | ツールの成功をどう測るか（利用者数？投稿数？応答時間？）が定義されていない。OSS公開後のGitHub Stars、Issue対応SLA等も未検討 |

### 2.2 長期的視点での懸念

- **メンテナンス持続性**: 個人プロジェクトがOSS化された場合、Issue対応・PR レビュー・バージョンアップ追従の体制は？
- **Slack API変更リスク**: レート制限変更（2025年5月の変更）に言及しているのは良いが、Slack APIの非推奨化・仕様変更への中長期的な対応方針が欲しい
- **MCP仕様の進化**: MCP自体がまだ発展途上のプロトコル。仕様変更への追従コストの見積もりがない

## 3. ドキュメント間の戦略的一貫性

### 3.1 良い点

- requirements.md → architecture.md への技術選定の流れが論理的
- architecture.md のADR（設計判断記録）が意思決定の透明性を確保
- slack-app-setup.md が requirements.md の「初期設定がわかりやすい」要件を具体化

### 3.2 改善すべき点

| # | 指摘 | 対象 | 説明 |
|---|---|---|---|
| S-5 | **ドキュメント間の相互参照が不足** | 全体 | 各ドキュメントが独立しており、読者がどの順序で読むべきかの導線がない |
| S-6 | **ステークホルダー定義が不在** | requirements.md | 「誰のため」のツールなのかが曖昧。開発者個人？チーム？OSS コミュニティ？段階ごとにターゲットが変わるはずだが未整理 |
| S-7 | **フェーズ分けが不在** | requirements.md | MVP→チーム展開→OSS公開の段階が未定義。全機能を一度に実装するのか、段階的に拡張するのかが不明 |

## 4. 推奨アクション（優先度順）

1. **【高】S-2: 競合分析と差別化ポイントの明記** — 既存Slack MCPツールとの比較表を追加し、「なぜ自作するのか」を1段落で説明
2. **【高】S-1: OSS公開の目的と判断基準の明記** — 公開の動機、最低品質基準、メンテナンス体制の方針を追記
3. **【中】S-7: 開発フェーズの定義** — Phase 1（MVP/個人利用）→ Phase 2（チーム展開）→ Phase 3（OSS公開）のロードマップを追加
4. **【中】S-6: ターゲットユーザーの明確化** — 各フェーズでの想定ユーザーペルソナを定義
5. **【低】S-5: ドキュメント読解ガイドの追加** — docs/README.md 等でドキュメント構成と読む順序を案内
